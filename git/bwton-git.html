# Step GitFlow

# 项目阶段：

1. 开发阶段
2. 测试阶段
3. 发布阶段
4. 线上bug阶段

# 基础分支：

Master
develop
feature*（多个）
hotfix
release (可以暂不考虑)

# git流程：

1. 项目迭代开始时由组长从develop拉取新的feature分支，进入需求开发。
2. 开发完成，使用feature进行自测，并发布到开发环境进行联调。
3. 自测联调完成，可以直接使用featrue，进入测试环境发版（或者合入develop，进入测试环境发版，由于develop分支的唯一性，当多个需求重叠时，可能会有冲突）
4. 测试完成后将feature合并到develop，由组长或指定小组长完成。
5. 生产(预发)时develop分支合并到master分支，此时如果存在bug，需在原来的feature分支中进行修改测试通过后再合并到develop分支中，再由develop合并到master分支。
6. 发版完成，验证通过，打tag，并通知其他正在feature开发的同事，拉取最新的develop到自己feature开发分支中，保证feature代码始终是基于develop最新代码开发。
7. 如果遇到生产BUG，需要从master拉取hotfix进行修正，测试通过后合并到master和develop中；如有进行中的feature，需要讲develop合到feature中

# 注意点：

1. master权限只有组长有
2. 不可在develop进行代码修改，所有修改都需要通过feature和hotfix来完成
3. feature从develop拉取
4. feature测试完成没有问题才能合并到develop
5. master上生产后，打tag(特性+日期)
6. 开发人员可以每日定期从develop拉取最新的代码到本地feature中 （暂定）

# 分支命名方式：

feature/v1.0.0/需求特性
hotfix/v1.0.0/需求特性+bug简述
