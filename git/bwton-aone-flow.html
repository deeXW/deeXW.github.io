# Step AoneFlow

# 项目阶段：

1. 开发阶段
2. 测试阶段
3. 发布阶段
4. 线上bug阶段

# 基础分支：

Master
feature*（多个）
hotfix
release/* (比如release/test。当然也可以直接使用test分支，具体看情况)

# aone流程(改造)：

1. 项目迭代开始时由组长从Master拉取新的feature分支，进入需求开发。
2. 开发完成，使用feature进行自测，并发布到开发环境进行联调。
3. 自测联调完成进入测试环节，从主干(master)上拉出一条新分支(release/test)，将所有本次要集成或发布的特性分支(featrue)依次合并过去，从而得到发布分支。(可以理解为，模拟主干的一次合并)
4. 测试过程如果存在bug，需在原来的feature分支中进行修改，测试通过后再合并到(release/test)分支中，再进行测试，不允许在release上修改代码。
5. 测试完成后将(release/*)合并到master，进行发布，由组长或指定小组长完成。
6. 发版完成，验证通过，打tag，并通知其他正在(feature)开发的同事，拉取最新的(master)到自己(feature)开发分支中，保证feature代码始终是基于(master)最新代码开发。
7. 如果遇到生产BUG，需要从(master)拉取(hotfix)进行修正，测试通过后合并到(master)；如有进行中的(feature)，也需要将最新代码合到(feature)中。

# 注意点：

1. (master)权限只有组长有
2. 不可在(master)进行代码修改，所有修改都需要通过(feature)和(hotfix)来完成
3. (feature)从(master)拉取
4. (release/*)测试完成没有问题才能合并到(master)
5. (master)上生产后，打tag(特性+日期)
6. 同一个仓库代码上线后，如果存在进行中的(feature)特性，需要通知各负责人，更新代码。

# 分支命名方式：

feature/v1.0.0/需求特性
hotfix/v1.0.0/需求特性+bug简述
tag/需求特性/日期
